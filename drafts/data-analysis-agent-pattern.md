# データ分析エージェントの設計パターン（素材）

## ステータス: 素材・構想段階

## コンセプト

データ分析エージェントの決定版パターン。4つの要素が1つの思想で貫かれている。

**思想: LLMから仕事を剥がす。LLMには「読んで、判断して、伝える」だけをさせる。**

## 4つの要素

### 1. レポート化アーキテクチャ

- 原子データを分子レベルに集約し、クエリの起点を変える
- LLMにデータ分析をさせない。ETLパイプラインが事前にデータを分析し、構造化レポートに変換する
- LLMはレポートを「読む」だけ
- 同じレポートを読めば同じ事実が出る。品質の制御ポイントがLLMの出力ではなくレポート選択に移る

詳細: design第6章、implementation第6章

### 2. タグによる二段階検索（tag as document）

- 番組に紐づくあらゆる属性（出演者、コーナー名、ジャンル、曜日、放送時間帯）を「タグ」として抽出
- タグをDBのラベルではなく、検索可能なドキュメントとして検索エンジンに格納
- 曖昧な入力→タグ検索→正確な検索条件→レポート検索、の二段階で正しいレポートに到達
- 曖昧さの解決をLLMではなく検索エンジンが担う

実装: report_agentのsearch_tags → search_program_schedules → search_reports → get_report

### 3. Instruction Paging Pattern

- レポート本文をstateに格納し、instructionに動的展開する
- events（会話履歴）を汚染しない
- temp:プレフィックスで自動ページアウト
- LLM自身がどのレポートを読むか、何件ずつ読むかを判断する
- State=ディスク、Instruction=RAM、LLM=CPU

詳細: agent-design/drafts/instruction-paging-pattern.md（別途保存）

### 4. Agent Probe（多層品質保証）

- 4層検証: 動いたか → 伝わったか → 安定しているか → 正しいソースを使ったか
- 一貫性スコア: 同じ質問をN回投げて統計で見る
- state_constraints: 使ったレポートを正規表現で静的検証
- TurnExecutor Protocolで全経路（local/agent/poll/longpoll/sse/bff）を共通化
- インストラクション変更 → テスト起動でTDD。統計的な品質チェック

実装: e2eスイート

## なぜこの4つが必要か

| 要素 | 解決する問題 | LLMから剥がしている仕事 |
|------|------------|---------------------|
| レポート化 | 推論の揺れ、コスト、レイテンシ | データ分析・集計 |
| タグ検索 | 曖昧入力の不安定な解決 | 入力の正規化・エンティティ解決 |
| Instruction Paging | コンテキスト汚染、attention希薄化 | メモリ管理 |
| Agent Probe | 品質保証のブラックボックス化 | 品質判定（第4層はLLM不要） |

## どれか1つが欠けると

- レポート化なし → LLMが毎回分析、結果がブレる
- タグ検索なし → 正しいレポートに到達できない、LLMのプロンプトで曖昧さを吸収しようとして不安定
- Instruction Pagingなし → 大量レポート読み込みでコンテキスト汚染、品質劣化
- Agent Probeなし → インストラクション変更の影響が定量的にわからない、手動確認に依存

## ドメイン依存性

構造自体はドメイン依存ではない。視聴率分析で検証済みだが、以下にも適用可能:
- 金融データ分析（銘柄→タグ、決算レポート→レポート化）
- マーケティング分析（キャンペーン→タグ、KPIレポート→レポート化）
- 医療データ分析（患者属性→タグ、検査結果→レポート化）
- 製造業の品質管理（製品→タグ、検査レポート→レポート化）

## 世の中との比較

- Anthropic「Building Effective Agents」→ 「シンプルにしろ」で止まる。具体的な分析エージェントのパターンは提示していない
- OpenAI Cookbook → ReActループが基本。分析はLLMの汎用能力任せ
- Manus Context Engineering → 「コンテキストを管理せよ」。方向性は同じだが、Instruction Pagingほど具体的ではない
- LangChain/LangGraph → フレームワークのパターンは提供するが、ドメイン固有の品質保証パターンは扱わない

## 実スケール

- 2,400以上の番組
- 1放送回あたり17種類のレポート
- 71,000 report_key以上
- 毎日データ更新
- 本番運用中（Google ADK + Agent Engine + A2A）
