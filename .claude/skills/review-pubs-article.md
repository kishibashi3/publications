# pubs 記事レビュースキル

「思考の本棚」の記事を、これまでの編集経験から蓄積された criteria で構造的にレビューする。

## スコープ

- **対象**: `docs/` 配下の `.md` ファイル（公開対象）のみ
- **対象外**: `drafts/` は保留領域なのでデフォルトでスキップ
- **明示的指示があれば例外**: ユーザーが `drafts/foo.md` を直接指定した場合のみ drafts/ もレビュー
- 「全記事をレビュー」「すべての記事を見直して」のような曖昧指示の場合は **`docs/` 配下のみ** を対象とする

## レビュー前の準備

1. **CLAUDE.md を読む** — Core of Core, Site Philosophy, Content Criteria の確認
2. **対象記事を Read** で全文取得
3. 記事が引用・依拠している **Core 章があれば該当章も読む**（特に ch.4 温度、ch.6 システム価値、supplementary/emergence-theory.md, chaos-edge-optimization.md）
4. 関連する subject 章があれば横断的に読む（用語のブレ確認）

## チェックリスト

### A. 還元主義チェック（最重要）

サイトの基盤は一般システム論。**全体は部分の和ではない**。

- [ ] システムを「層」「成分」に線形分解していないか?（例: 政治層/社会層、自然/人工、コア/周辺の単純分離）
- [ ] 「全体 = 部分の和」を暗に仮定する記述がないか?
- [ ] decomposition で説明できないものを decomposition で説明していないか?
- [ ] 共発進化（coevolution）する要素を独立変数として扱っていないか?

→ 違反があれば **「これは whole として扱うべき」** と指摘し、書き直し案を出す。

### B. ヘッジ表現チェック

CLAUDE.md「主張は確信的に断言する」に違反していないか。

- 検出語: 「〜と思う」「〜と考えている」「〜かもしれない」「〜可能性がある」「〜だろう」「〜と思われる」「〜気がする」「〜と言える」
- 例外: **検証不能な推測**は推測として明示してよい
- 違反 → 断言形に書き換え提案

### C. フレームワーク整合性チェック

Core の中心命題と矛盾していないか。

- [ ] **適温（50°C 付近）で創発が最大化** — 温度と創発の対応が単峰性に従っているか
- [ ] **価値 = 創発 × 存続期間** — 単独の指標で判定していないか
- [ ] **温度勾配 ∇T が流れの源** — 勾配ゼロで創発が起きる主張になっていないか
- [ ] **倫理 = 温度制御の技術** — 倫理が善悪論として書かれていないか
- [ ] 数値・表・図示が Core 方程式と整合しているか?

数値矛盾の例: 「米国 75°C」かつ「創発極めて高い」は適温ピーク仮定と矛盾する。

### D. 自己矛盾チェック

- [ ] 同一記事内で前段と後段が矛盾していないか?
- [ ] 「Xは問題ではない」と言いつつ X で診断していないか?（実例: 「温度自体は問題ではない」と書きながら温度で各国を診断）
- [ ] 用語の意味が記事内でブレていないか?

### E. 表現の強度チェック

CLAUDE.md「攻撃的表現は構造の観察に変換する」。

- 検出語: 「覚悟がない」「使い切る」「割り切る」「甘い」「弱い」「べき」「すべきだ」「正しい/間違っている」
- これらは価値判断を含む。**構造の機能描述**に書き換え可能か検討する
- 例: 「地方を捨て場として使い切る覚悟がない」→「地方を経済的に放棄する選択をしていない」
- 断言の強度は維持しつつ、価値判断ではなく機構記述に。

### F. 抽象階層チェック

Core が抽象層、subject 章は適用層。

- [ ] 新しい概念を導入する前に Core 既出か確認したか?
- [ ] 熱力学標準用語（温度、勾配、エントロピー、熱機関、捨て場）を「新概念」として再定義していないか?
- [ ] subject 章固有の中級概念を Core レベルに格上げしすぎていないか?
- [ ] 同じ概念に複数章で複数の名前が付いていないか?（→ 統一すべき）

### G. 「全員幸せな花畑」チェック

CLAUDE.md philosophy に沿って、win-win 思考の盲信を検出する。

- [ ] 「全員が勝つ」「誰も損しない」「win-win」「平等」を **無条件に** 肯定していないか?
- [ ] それが創発エンジンの破壊（勾配の消失）と矛盾しないか?
- [ ] 「公平 / 均等 / 全員救う」が結論になっている記述は再検討
- 結論として 「**全員幸せな花畑は、創発力を失って滅びる**」（社会温度論）と整合するか確認

### H. 外部依存の見落としチェック

「自立的」「均質」「勾配なし」と書いた対象が、本当にそうか。

- [ ] 国・組織・システムを **内部だけで論じて** 外部依存（offshoring, 資源輸入, グローバル供給網）を見落としていないか?
- [ ] 「勾配がない」と書いた対象は、外部勾配を持っている可能性を検討したか?
- 例: 「北欧は勾配なし」→ 実際は外部勾配（offshoring・北海油田等）あり

### I. 時間軸チェック

- [ ] 現在の状態だけで論じて、過去・未来の異なる状態を見落としていないか?
- [ ] 同じシステムでも時代によって状態が変わる事実を反映しているか?
- 例: 「日本は勾配を否定する」→ 高度経済成長期は明確に勾配運用していた

### J. 個人体験 / 過学習チェック

- [ ] 個人的体験（パワハラ、引きこもり、特定の職場体験等）からの一般化になっていないか?
- [ ] 攻撃的・怨念的トーンが残っていないか?
- [ ] 「私の経験では」「ある会社では」のようなアネクドート起点の論証になっていないか?

### K. サイトトーンチェック

- [ ] 「研究」「論文」「学術的」を前面に出していないか?
- [ ] 章タイトル・セクション名が冗長になっていないか?（章番号 + 短いタイトルが basis）
- [ ] サイドバーやナビゲーション設計に「圧」がないか?
- [ ] 「思考」「考えたこと」のトーンを保てているか?

### L. Lint / Build チェック

機械的なチェックも回す:

- [ ] `npm run lint` が通る（textlint + link-check）
- [ ] `npm run docs:build` が通る
- [ ] config.js のサイドバーに該当ファイルが登録されているか? （新規記事なら）

## 出力形式

レビュー結果は以下の形式で:

```
## レビュー対象
{path}

## OK な点
- {checkpoint をクリアしている部分}

## 指摘事項

### {critical / major / minor} : {チェック項目}
- 場所: {file}:{line}
- 現状: {問題のある記述}
- 指摘: {何が問題か}
- 提案: {具体的な書き換え案}

## 総合判断
{publish 可 / 修正後 publish / 大幅書き直し} の判定 + 理由
```

## 運用上のポイント

- **A（還元主義）と C（framework 整合）と G（win-win）が最頻発** の指摘ポイント。重点的に見る
- 修正提案は **断言の強度を保ったまま** 行う。ヘッジを入れて柔らかくしない
- ユーザの指摘で発見されたパターンは、このスキルに **追記して育てる**
- レビューは publish 前のセルフチェックとしても、publish 後の見直しとしても使える

## このスキルの由来

2026-05-09 の社会温度論編集セッションで著者から繰り返し受けた指摘を集約した。具体的には:

- 還元主義への警告（自然/人工分解、政治層/社会層分解 への push back）
- フレームワーク矛盾の指摘（米国 75°C で創発極めて高いは適温50ピークと矛盾）
- ヘッジの修正（捉えたい → 捉えなければならない）
- 強表現の調整（覚悟がない、使い切る を構造観察に変換）
- 外部依存の見落とし指摘（北欧の外部勾配）
- 時間軸の指摘（高度経済成長期は heat engine だった）
- 抽象階層の指摘（熱力学標準用語を新概念ページにしない）
